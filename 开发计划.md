## 实际数据链路（源码分析结论）

```
bilibili-monitor                refine_batch.py               relationship-analysis
(下载+Whisper转写→原始txt)  →  (LLM精炼三段式+LLM分类)  →   (BM25+向量语义检索+LLM问答)
输出：~/B站监控/<UP>/xxx.txt   输出：情感素材库/<分类>/xxx.txt   存储：references/情感素材库/ 1413个文件
```

### 关键发现
- **llm-summarizer = refine_batch.py**：架构文档中的"LLM精炼"就是这个脚本，调 MiniMax/deepseek API
- **精炼格式**：三段式（核心观点 + 案例摘要 + 可行动建议）
- **分类体系**：31个情感/两性分类，由 LLM 自动分类
- **relationship-analysis** 已有 1413 个精炼文件 + 完整语义搜索引擎
- **bilibili-monitor** 输出原始txt，refine_batch.py 消费并精炼，两者通过文件系统串联

## 开发顺序建议

```
Phase 1 (bilibili-monitor)     ← 数据源头，最先改造
    ↓
Phase 3 (Text-to-SQL)          ← 改 DuckDB schema，依赖 Phase 1 的表结构
    ↓  （可并行）
Phase 2 (RAG)                  ← 改 ChromaDB，依赖 Phase 1 的写入格式
    ↓
Phase 4 (Router Agent)         ← 依赖 Phase 2 + 3 的 API
    ↓
Phase 5 (统一前端)              ← 依赖 Phase 4 的 API
    ↓
Phase 6 (Docker Compose)       ← 所有服务就绪后统一编排
    ↓
Phase 7 (串联调试)             ← 最终验收
```

Phase 1 → Phase 2/3 可部分并行，Phase 4/5 必须串行。
