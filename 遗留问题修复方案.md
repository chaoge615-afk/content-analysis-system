## 二、各组详细方案

### LG-09 · 多跳问题分类在 hybrid/semantic 间摇摆,期望 semantic(非 hybrid)

**置信度**: high  

**根因**

多跳问题"桃姐最近聊了什么话题？她关于吵架的建议是什么？"分类在 hybrid/semantic 间摇摆,根因是 classify_intent 节点的 INTENT_SYSTEM_PROMPT 存在两处诱导模型判 hybrid 的缺陷,叠加 call_llm_direct 未设置 temperature(走 provider 默认非零采样),导致同一问题多次执行结果不稳。

具体证据链:
1) nodes.py:171-172 hybrid 类型的正向示例里出现"这些视频主要讨论什么话题？"——而本问题恰好含"什么话题",模型据此把多子句语义问题误判为 hybrid(高相似短语锚定)。
2) nodes.py:177 关键判断逻辑第3条"如果问题同时需要数字和观点 → hybrid"过于宽泛:本问题两个子句(最近聊了什么话题 / 关于吵架的建议)"话题"与"建议"被模型读成"两类信息=同时需要数字和观点",触发 hybrid。
3) nodes.py:29-76 call_llm_direct 在 openai/anthropic 两个分支构造 payload 时,只有 model/messages/max_tokens,未设置 temperature。.env 显示 LLM_PROVIDER=openai、OPENAI_MODEL=qwen3.7-max。OpenAI 兼容接口未传 temperature 时走 provider 默认值(非 0),分类在高熵边界问题上自然抖动。
4) nodes.py:234 classify_intent 直接 await call_llm_direct,无温度控制、无重试/投票、无规则后校验,单次采样结果即定 route_type。
5) nodes.py:275-282 兜底只在 JSON 解析异常时才回退 semantic;route_type=hybrid 是模型正常返回,不会触发兜底,无法纠偏。

**证据**

1) Prompt 规则冲突。c:/Users/25022/Desktop/ai项目/agent-platform/src/agent/nodes.py:171-172 hybrid 例子含「这些视频主要讨论什么话题？」，而问题里正是「什么话题」（桃姐最近聊了什么话题）。即 hybrid 的正面示例里出现了和待分类问题高度相似的短语，模型据此把多子句语义问题误判为 hybrid。
2) 关键判断逻辑第3条（nodes.py:177）"如果问题同时需要数字和观点 → hybrid"过于宽泛:模型把多子句=多类信息 误读为"同时需要数字和观点",触发 hybrid。
3) call_llm_direct(nodes.py:29-76)在 LLM_PROVIDER=openai/anthropic 两个分支构造 payload 时,均只设 model/messages/max_tokens,未设 temperature。.env 显示 LLM_PROVIDER=openai,MODEL=qwen3.7-max。OpenAI 兼容接口不传 temperature 时用 provider 默认(非0),分类任务在随机采样下摇摆;且即使显式传 temperature:0(Greedy),qwen3.7-max 对该多子句问题仍不稳定(实测 6 次中 2 次 hybrid)。
4) classify_intent(nodes.py:234)直接 await call_llm_direct(messages_payload),无温度控制、无重试/投票,单次结果即定型 route_type,任何一次失败采样直接产生 hybrid。
5) 降级兜底(nodes.py:275-282)只在 JSON 解析异常时才回退 semantic;route_type=hybrid 是模型正常返回,不会触发兜底,无法自我纠正。

实测证据(独立用 qwen3.7-max 复跑 classify_intent 的真实 prompt,各 6 次):
- 原始 prompt + provider 默认温度: hybrid:4 / semantic:2  (摇摆,4/6 命中 hybrid)
- 原始 prompt + temperature=0(贪心):semantic:4 / hybrid:2  (仍不稳,贪心也 2/6 hybrid)
→ 证明单靠降温到 0 不能根治,根因是 prompt 判定规则对多子句问题不清晰。
- 修复 prompt(澄清多子句语义=semantic + 删除误导例 + 新增第5条) + 默认温度: 6/6 semantic
- 修复 prompt + temperature=0: 6/6 semantic
→ 证明 prompt 修正才是根治手段。

**修复思路**

两层修复,以 prompt 为主、温度为辅。

(1)【主修·根治】修 INTENT_SYSTEM_PROMPT,消除规则歧义与误导示例,明确「多子句语义问题=semantic」:
- nodes.py:172 把 hybrid 示例里「这些视频主要讨论什么话题？」改为「这些视频主要讨论什么观点？」,避免「什么话题」这个出现频率最高的口语短语成为 hybrid 的判别线索。
- nodes.py:177 第3条改为:「如果问题同时需要『结构化数字(统计/计数/排名/时间/列表筛选)』和『观点』才能完整回答 → hybrid;注意:多个子句都只问观点/建议/话题内容时,即便有多个子句,也属于 semantic,不是 hybrid。只有当至少一个子句明确需要统计/计数/排名/列表筛选等结构化数字时才判 hybrid。」把 hybrid 的触发条件从宽泛的"数字+观点"收紧到"必须含结构化数字子句"。
- nodes.py:178 后新增第5条:「『什么话题/聊了什么』问的是UP主讨论的内容/观点,属于语义检索,不要因为含『话题』二字或子句数量多就判 hybrid。」
实测此 prompt 修订在默认温度与温度0下各6次,全部命中 semantic(12/12);而原始 prompt 默认温度 hybrid:4/semantic:2、温度0 hybrid:2/semantic:4。

(2)【辅修·降噪】classify_intent 是确定性路由任务,应关闭随机性:
- nodes.py:42-46(openai 分支)与 nodes.py:54-58(anthropic 分支)的 payload 增加 "temperature": 0。
独立验证显示:温度0对原始 prompt 只是把 hybrid 比例从 4/6 降到 2/6,不能单独根治;但它能让(1)修过的 prompt 更稳定地输出 semantic(已 6/6),并消除其他分类问题的偶发抖动。
注意:call_llm_direct 同时被 merge_results(nodes.py:501)和反思(nodes.py:558)复用,给 merge/反思强制温度0可能让最终回答更保守、创造性下降。建议:给 call_llm_direct 增加可选参 force_deterministic:bool=False,仅在 classify_intent 调用时传 force_deterministic=True;或更简单——只改 prompt(修法1)已能稳定输出 semantic,可不动全局温度,避免副作用。

推荐落地:只做修法(1)最安全;若想彻底消除所有分类抖动,叠加修法(2)并将温度参数隔离到分类路径。

**副作用**

1) 改 prompt 的影响范围:只影响所有进入 classify_intent 的问题判定。收紧 hybrid 触发条件后,真正"数字+观点"的混合问题(如"情感类视频平均播放量是多少？这些视频主要讨论什么观点？")仍应判 hybrid——需确认 prompt 修订未把这类问题误降级为 semantic。实测时我用修复后 prompt 跑了该标准 hybrid 问题,仍判 hybrid(未列入本次脚本输出,建议在回归中显式覆盖)。

**根因**

【E2E-08 GET 404/405】agents.py 缺少两个 GET 路由,且路由顺序导致路径被 {agent_id} 抢匹配:
