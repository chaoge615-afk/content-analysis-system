## 一、项目概述

一套从 B站视频采集到智能问答的端到端 AI 系统。

**核心链路**：B站情感博主视频 → 自动下载转写 → LLM 精炼 → 结构化入库(DuckDB) + 向量化入库(ChromaDB) → 统一入口智能问答

**作者**：李畅超
**GitHub**：https://github.com/chaoge615-afk/content-analysis-system

## 三、数据采集工作流

### 3.1 bilibili-monitor 全流程

```
[Cron 触发 · 每6小时 · bilibili-cron 容器]
    ↓
monitor_all.py（入口脚本）
    ├── 扫描 config/*.yaml（UP主配置）
    ├── 分批执行（MAX_CONCURRENT_TRANSCRIBE=2）
    └── 每个 UP 主调用 monitor.py
         ↓
    monitor.py（单个UP主处理）
         ├── 1. get_up_videos.py → WBI API 拉取视频列表
         ├── 2. download_videos.py → yt-dlp 下载 m4a
         ├── 3. transcribe_local.py → faster-whisper 转写 → txt
         ├── 4. refiner.py → DeepSeek API 精炼 → 三段式摘要
         ├── 5. db_writer.py → 写入 DuckDB（video_meta 表）
         ├── 6. chroma_writer.py → 写入 ChromaDB（video_knowledge）
         └── 7. qq_notify.py → QQ Bot 推送通知
```

## 四、数据查询工作流

### 4.1 意图分类

```
用户问题
    ↓
  路由Agent（router-agent :8000）
    ├── 调用 MiniMax M2.7 做意图分析
    ├── 判断类型：structured / semantic / hybrid
    ├── 提取过滤条件：
    │   ├── up_name（UP主名称）
    │   ├── category（只能用 31 个有效分类名）
    │   ├── date_range（时间范围）
    │   └── keywords（话题关键词）
    └── 输出：路由决策 + 过滤条件
```

**重要：** category 是目录分类名（如 `01_喜欢`），不是话题关键词。"冷暴力"是话题（放 keywords），不是分类。

### 4.2 三种查询模式

```
① structured（结构化查询）→ Text-to-SQL
   - 涉及数量统计、时间排序、比较等
   - 示例："桃姐最近发了几个视频？"
   - 流程：问题 → SQL 生成 → DuckDB 查询 → 返回结果

② semantic（语义查询）→ RAG
   - 涉及观点、建议、内容理解等
   - 示例："博主们对冷暴力怎么看？"
   - 流程：问题 → BM25+向量混合检索 → metadata 过滤 → LLM 生成回答

③ hybrid（混合查询）→ SQL + RAG 并行
   - 同时包含结构化和语义需求
   - 示例："桃姐最近聊了什么？她关于吵架的建议？"
   - 流程：SQL 先筛 + RAG 检索 → LLM 融合回答
```

### 4.3 RAG 检索流程

```
用户问题
    ↓
  BM25 关键词检索（rank_cap=3）
    +
  向量相似度检索（top_k=5, SiliconFlow Embedding）
    ↓
  混合排序融合（HybridSearch）
    ↓
  metadata 过滤（up_name / category / bvid）
    ↓
  拼接上下文（附带来源信息：UP主、分类、BV号）
    ↓
  LLM 生成回答
  ├── DeepSeek V4 Pro（默认）
  └── MiniMax M2.7（备选）
```

## 五、部署环境

### 5.1 硬件环境对比

| 配置 | 本地电脑（开发） | NAS（生产） |
|------|-----------------|-------------|
| CPU | AMD Ryzen 7 8845H（8核16线程） | Intel N150（4核低功耗） |
| 内存 | 24GB | DDR4 8G（模块化，可扩展） |
| GPU | NVIDIA RTX 4060 Laptop（8GB 显存） | 无独显（集成核显） |
| Whisper | GPU 加速，秒级转写 | faster-whisper CPU，3-5分钟/条 |
| 用途 | 开发调试，下班后使用 | 7×24 生产运行，上班时远程开发 |
| Docker | Docker Desktop（WSL2 后端） | Docker Compose 统一编排 |

### 5.2 开发策略

```
           GitHub（中心仓库）
          ↗                    ↘
    本地电脑                NAS
  （下班后开发）          （上班时远程开发 / 生产运行）
  RTX 4060 · 24G         N150 · 8G
  GPU Whisper            CPU Whisper
```

核心原则：
- 所有代码必须经过 GitHub，禁止两边各自改不提交
- 环境差异走配置文件，不走代码分支
- 每天开工前 pull，收工前 push

### 5.3 Docker Compose profiles

```bash
# 开发环境
docker compose --profile dev up -d

# 生产环境（含定时调度）
docker compose --profile nas up -d

# 按需运行采集
docker compose run --rm bilibili-monitor
```

## 六、版本管理与开发流程

### 6.1 Git 分支策略

```
master        ← 稳定版本（NAS 生产环境跑这个）
dev           ← 日常开发（本地和NAS都在这个分支上改）
feature/xxx   ← 大功能单独拉分支（可选）
```

### 6.2 环境差异处理

代码层面完全一致，差异只体现在配置上：

#### `.env` 文件（不提交到 Git）

```env
# 本地 .env（开发环境）
WHISPER_MODEL=large-v3
WHISPER_DEVICE=cuda
COMPUTE_TYPE=float16

# NAS .env（生产环境）
WHISPER_MODEL=small
WHISPER_DEVICE=cpu
COMPUTE_TYPE=int8
```

#### `.gitignore` 必须排除

```gitignore
.env
dev-data/
*.pyc
__pycache__/
venv/
node_modules/
```

### 6.3 每日工作流

```
上班前（本地）：
  git add → git commit → git push
  关掉本地，去上班

上班时（远程到NAS）：
  git pull
  改代码 → git commit → git push
  NAS 验证：docker compose --profile nas up -d

下班后（本地）：
  git pull（拉 NAS 上的改动）
  继续开发 → git commit → git push
  本地验证：docker compose --profile dev up
```

### 6.4 防冲突习惯

```
① 开工第一件事：git pull --rebase
② 改完一个功能就 commit，别攒着
③ 别两边同时改同一个文件
④ 如果本地用 Claude Code 改，NAS 上也用 Claude Code，确保不在同一个文件上工作
```

## 九、注意事项

```
- N150 转写时 CPU 占用高，串行处理，不要并发
- B站 Cookie 会过期（code=-352），需监控失败并 QQ 通知
- 转写完成后自动删除 m4a，只保留 txt
- faster-whisper small 约需 2-3GB 内存，设 Docker 内存上限
- 双层 Checkpoint 防呆：downloaded.txt（已下载）+ done_bvid.txt（已转写完成）
- text-to-sql 声称用 CrewAI 但实际是纯 Python 手写 pipeline
- bilibili-monitor 已从 Hermes Skill 改造为独立服务，无 Hermes 依赖
- 意图分类器 category 只能用 31 个有效分类名，话题词放 keywords
- Docker 构建使用国内镜像加速（apt: 阿里云, pip: 清华 TUNA, npm: 淘宝）
- 每次镜像重建后执行 docker builder prune -a -f 清理缓存

内存管理（DDR4 8G）：
- 常驻服务（chromadb + FastAPI 服务 + nginx + cron）：约 6.4G
- bilibili-monitor 仅按需启动，跑完自动退出，释放 4G
- 转写期间峰值约 7-8G，在 8G 范围内可接受
- NAS 支持模块化内存扩展，未来可升级至 16G+
```
