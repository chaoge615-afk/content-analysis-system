## 一、项目概述

一套从 B站视频采集到智能问答的端到端 AI 系统，由三个独立项目组成，最终目标是串联为一个统一系统。

**核心链路**：B站情感博主视频 → 自动下载转写 → LLM 生成摘要 → 结构化入库(DuckDB) + 向量化入库(ChromaDB) → 统一入口智能问答

**作者**：李畅超
**GitHub**：https://github.com/chaoge615-afk

## 二、当前状态（已完成）

### 项目1：B站UP主监控转写系统（bilibili-monitor）

- **状态**：✅ 已完成，稳定运行 2-3 个月

- **技术栈**：Python + B站 WBI API + yt-dlp + Whisper (GPU) + QQ Bot + Cron + Hermes Agent Skill

## 三、待建设（规划中，尚未开发）

### 3.1 串联三个项目

目前三个项目相互独立，需要串联为统一系统：

#### 第1步：B站监控输出结构化数据到 DuckDB（1-2天）

bilibili-monitor 转写完成后，将结构化元数据写入 DuckDB：

```sql
video_meta 表:
  bvid TEXT PK, up_name TEXT, up_uid TEXT, title TEXT,
  publish_date DATE, category TEXT, duration INT,
  summary TEXT, tags TEXT, created_at DATETIME

up_info 表:
  uid TEXT PK, name TEXT, total_videos INT,
  last_update DATE, config_file TEXT
```

这样 Text-to-SQL Agent 可以查询视频元数据：
- "桃姐最近发了几个视频？"
- "哪个博主更新最频繁？"

#### 第2步：RAG 知识库支持 metadata 过滤（2-3天）

ChromaDB 存储时带上元数据，支持按博主/分类/时间过滤检索：

```
collection: "video_knowledge"
  document:  转写全文 / 摘要文本
  embedding: SiliconFlow Embedding 向量
  metadata:
    source: "bilibili"
    bvid: 视频BV号
    up_name: UP主名称
    title: 视频标题
    category: 内容分类
    publish_date: 发布日期
    chunk_index: 分块序号
    content_type: "full" / "summary"
```

#### 第3步：路由 Agent + 统一前端入口（3-5天）

新增 router-agent 服务，用 LLM 做意图识别，自动分发查询：

| 查询类型 | 示例 | 路由目标 |
|----------|------|----------|
| structured | "桃姐最近发了几个视频？" | → Text-to-SQL → DuckDB |
| semantic | "博主们对冷暴力怎么看？" | → RAG → ChromaDB |
| hybrid | "桃姐最近聊了什么？她关于吵架的建议？" | → SQL先筛 + RAG再检索 → LLM融合回答 |

### 3.2 各项目独立优化

#### Text-to-SQL 优化
- 建立评测指标（用测试集输出准确率、执行成功率）
- 错误自修复（SQL审查Agent发现错误后反馈给生成Agent重试）
- Few-shot 动态检索（从历史案例库自动检索相似示例）
- LangGraph 重写版本（GitHub 做 langgraph 分支，证明双框架能力）
- 对话记忆（支持多轮对话上下文）

#### RAG 系统优化
- 混合检索（向量检索 + BM25 关键词检索，结果合并）
- Reranking 重排序（Cross-Encoder 或 LLM 对检索结果重排）
- RAGAS 评估体系（Faithfulness、Context Relevancy、Answer Relevancy）
- Query 改写（LLM 改写用户问题为更适合检索的 Query）
- 多格式文档支持（PDF/Word/PPT/网页）
- 替换向量数据库（Milvus 或 Qdrant 替换 ChromaDB）

#### B站监控优化
- Web 管理面板（FastAPI + React：UP主管理、转写状态、历史查询）
- 转写质量优化（对比不同 Whisper 模型效果）

## 五、版本管理与开发流程

### 5.1 Git 分支策略

```
main          ← 稳定版本（NAS 生产环境跑这个）
dev           ← 日常开发（本地和NAS都在这个分支上改）
feature/xxx   ← 大功能单独拉分支（可选）
```

NAS 生产只跑 main 分支，开发完成后 dev → PR → merge 到 main → NAS 上 `docker compose pull && up -d`

### 5.2 环境差异处理

代码层面完全一致，差异只体现在配置上：

#### `.env` 文件（不提交到 Git）

```env
# 本地 .env（开发环境）
WHISPER_MODEL=large-v3
WHISPER_DEVICE=cuda
COMPUTE_TYPE=float16

# NAS .env（生产环境）
WHISPER_MODEL=small
WHISPER_DEVICE=cpu
COMPUTE_TYPE=int8
```

#### Docker Compose profiles（同一份文件）

```yaml
services:
  bilibili-monitor:
    profiles: ["nas"]
    mem_limit: 3g
    environment:
      - WHISPER_DEVICE=cpu

  bilibili-monitor-dev:
    profiles: ["dev"]
    deploy:
      resources:
        reservations:
          devices:
            - capabilities: [gpu]
```

本地开发：`docker compose --profile dev up`
NAS 部署：`docker compose --profile nas up -d`

#### `.gitignore` 必须排除

```gitignore
.env
dev-data/
*.pyc
__pycache__/
venv/
```

### 5.3 每日工作流

```
上班前（本地）：
  git add → git commit → git push
  关掉本地，去上班

上班时（远程到NAS）：
  git pull
  改代码 → git commit → git push

下班后（本地）：
  git pull（拉 NAS 上的改动）
```

## 十、注意事项

```
- N150 转写时 CPU 占用高，串行处理，不要并发
- B站 Cookie 会过期（code=-352），需监控失败并 QQ 通知
- 转写完成后自动删除 m4a，只保留 txt
- faster-whisper small 约需 2-3GB 内存，设 Docker 内存上限
- 双层 Checkpoint 防呆：downloaded.txt（已下载）+ done_bvid.txt（已转写完成）

内存管理（DDR4 8G）：
- 常驻服务（chromadb + FastAPI 服务 + nginx）：约 3-4.5G
- bilibili-monitor 仅 Cron 触发时启动（docker compose run --rm），跑完自动退出，释放 3G
- 转写期间峰值约 5.5-7.5G，在 8G 范围内可接受
- 各服务 Docker Compose 内存上限参考：
  ├── bilibili-monitor: 3g
  ├── chromadb: 1g
  ├── rag-service: 512m
  ├── text-to-sql: 512m
  ├── router-agent: 512m
  └── frontend: 256m
- NAS 支持模块化内存扩展，未来可升级至 16G+
```
