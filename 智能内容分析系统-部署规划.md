| CPU | AMD Ryzen 7 8845H（8核16线程） | Intel N150（4核低功耗） |
| 内存 | 24GB | DDR4 8G（模块化，可扩展） |

## 三、Whisper CPU 转写优化策略

N150 算力有限，需要针对性优化：

### 3.1 模型选择

```
优先级（速度 vs 质量权衡）：

方案A：faster-whisper（推荐）
├── 基于 CTranslate2，CPU 速度比原版快 4-6 倍
├── small 模型：单条8分钟视频约 3-5 分钟转写
└── 中文效果可接受

方案B：原版 Whisper small
├── 单条8分钟视频约 15-20 分钟
└── 质量略好于 faster-whisper

方案C：云端 API（备选）
├── 调用阿里云/讯飞 语音转写 API
├── 速度最快，但需付费
└── 适合大量历史数据补转写
```

### 3.2 转写调度策略

```
每6小时触发一次（Cron）
    ↓
拉取新视频列表（通常 0-5 个）
    ↓
串行转写（避免CPU过载）
├── 单条视频：下载 → 转写 → 精炼 → 入库
├── 完成后处理下一条
└── 预计耗时：0-5个视频 × 5分钟 = 最多25分钟
    ↓
全部完成后发送QQ通知
```

### 3.3 已有 txt 文件迁移

```
大部分历史视频已转写为 txt，无需重新转写：

1. 将已有 txt 文件同步到 NAS（rsync/scp）
2. 跑一次 llm-summarizer 批量精炼 + 入库
   ├── 结构化元数据 → DuckDB
   └── 向量化内容 → ChromaDB
3. NAS 接管后续增量数据即可

预计耗时：精炼+入库 远快于转写，几十条数据几分钟搞定
```

## 四、完整数据流

### 4.1 数据写入流程

```
[Cron 触发 · 每6小时]
    ↓
bilibili-monitor
    ├── 1. WBI API 拉取各UP主新视频
    ├── 2. yt-dlp 下载音频（m4a）
    ├── 3. faster-whisper CPU 转写 → txt
    ├── 4. 写入 downloaded checkpoint
    └── 5. 调用 llm-summarizer
                ↓
          llm-summarizer
                ├── 6. Minimax API → 生成摘要
                ├── 7. 写入 DuckDB（结构化元数据）
                │      video_meta 表
                ├── 8. 写入 ChromaDB（向量+metadata）
                │      video_knowledge collection
                └── 9. 写入 done_bvid checkpoint
                          ↓
                    QQ Bot 推送通知
                    "新增3个视频，已转写并入库"
```

```
用户提问（Web前端）
    ↓
router-agent
    ├── 意图分析（Minimax API）
    ├── 提取过滤条件（up_name, category, date）
    └── 路由决策
         ↓
    ┌────┴────┐
    ↓         ↓         ↓
structured  semantic   hybrid
    ↓         ↓         ↓
text-to-    rag-      并行调用
sql-agent   service   ├── text-to-sql
    ↓         ↓       └── rag-service
 DuckDB   ChromaDB       ↓
  查询     检索        结果融合
    ↓         ↓         ↓
    └────┬────┘         │
         ↓              ↓
    router-agent  LLM融合回答
         ↓
    返回前端展示
```

## 六、Docker Compose 服务编排（结构示意）

```
# 使用 profiles 区分环境，同一份 compose 文件管两个环境

services:
  nginx:            反向代理（80/443）
  frontend:         React 前端（静态资源）               mem_limit: 256m
  router-agent:     路由Agent（FastAPI · 8000）          mem_limit: 512m
  text-to-sql:      Text-to-SQL Agent（FastAPI · 8001）  mem_limit: 512m
  rag-service:      RAG 服务（FastAPI · 8002）           mem_limit: 512m

  # NAS 生产版（CPU 转写）
  bilibili-monitor:
    profiles: ["nas"]
    mem_limit: 3g
    environment:
      - WHISPER_DEVICE=cpu

  # 本地开发版（GPU 转写）
  bilibili-monitor-dev:
    profiles: ["dev"]
    deploy:
      resources:
        reservations:
          devices:
            - capabilities: [gpu]

  llm-summarizer:   LLM 精炼（Python · 被monitor调用）   mem_limit: 512m
  chromadb:         向量数据库（8003）                    mem_limit: 1g

volumes:
  duckdb-data:      结构化数据
  chromadb-data:    向量数据
  bilibili-data:    B站原始数据
```

本地开发：`docker compose --profile dev up`
NAS 部署：`docker compose --profile nas up -d`

## 八、性能预估（N150 CPU）

```
单条视频转写（8分钟音频）：
├── faster-whisper small：约 3-5 分钟
├── faster-whisper base：约 1-2 分钟
└── 原版 whisper small：约 15-20 分钟

每次Cron触发（假设新增3个视频）：
├── 下载：约 2 分钟
├── 转写：约 10-15 分钟（faster-whisper small）
├── LLM精炼：约 1 分钟（API调用）
├── 入库：约 30 秒
└── 总计：约 15-20 分钟

查询响应时间：
├── Text-to-SQL：2-5 秒
├── RAG 检索：3-8 秒
└── 混合查询：5-12 秒

内存占用（DDR4 8G）：
├── 系统 + Docker 开销：~1-1.5G
├── 常驻服务（chromadb + FastAPI + nginx）：~3-4.5G
├── bilibili-monitor 转写期间峰值：~5.5-7.5G
├── bilibili-monitor 仅 Cron 时启动（docker compose run --rm），跑完自动退出
└── 余量充足，未来可扩展内存至 16G+
```

## 九、注意事项

```
⚠️ N150 转写时CPU占用较高
   → 转写期间暂停其他CPU密集型任务
   → 串行处理，不要并发

⚠️ B站Cookie会过期
   → 设置过期监控，失败时QQ通知提醒
   → 定期手动更新Cookie

⚠️ 音频文件占空间
   → 转写完成后自动删除 m4a 文件
   → 只保留 txt 转写文本

⚠️ Docker 内存限制（DDR4 8G）
   → faster-whisper small 约需 2-3GB 内存
   → bilibili-monitor 仅 Cron 触发时启动（docker compose run --rm），跑完自动退出释放内存
   → 常驻服务（chromadb + FastAPI + nginx）约 3-4.5G，在 8G 内可接受
   → 各服务设置 mem_limit 防止单个服务占用过多内存
   → NAS 支持模块化内存扩展，未来可升级至 16G+

⚠️ bilibili-monitor 启动方式
   → Cron 表达式：0 */6 * * *（每6小时）
   → 启动命令：docker compose run --rm bilibili-monitor
   → restart: "no"（不自动重启，仅按需启动）
```
