# 智能内容分析系统 - 整体架构

---

## 一、系统定位

一套从**非结构化视频内容**到**智能问答**的端到端 AI 系统：
- 自动采集 B站博主视频内容
- 转写 → LLM 精炼 → 结构化入库 + 向量化入库
- 统一入口：路由 Agent 自动分发到 Text-to-SQL 或 RAG 知识库
- 支持结构化查询、语义检索、混合查询三种模式

## 三、数据流架构

### 3.1 数据采集链路（写入）

```
B站情感博主视频
    ↓ [Cron · 每6小时]
  WBI签名API → 拉取新视频列表
    ↓
  yt-dlp → 下载音频（m4a）
    ↓
  faster-whisper CPU → 语音转文字（faster-whisper small · CTranslate2加速）
    ↓ [双层Checkpoint容错]
  LLM精炼 → 生成视频摘要
    ↓
  ┌─────────────────────────────┐
  │        分流写入              │
  │                             │
  ├─────────────┬───────────────┤
  ↓             ↓               ↓
DuckDB        ChromaDB      知识库Skill
结构化元数据   向量存储        原始文本归档
  │             │               │
  │  video_meta │  embeddings   │  transcripts
  │  up_info    │  + metadata   │
  └─────────────┴───────────────┘
```

### 3.2 数据查询链路（读取）

```
用户提问
    ↓
  路由Agent → 意图分析
    ↓
  ┌─────────────────────────────────────┐
  │          三种路由策略                 │
  │                                     │
  │  ① 结构化查询                        │
  │     "桃姐最近发了几个视频？"           │
  │     → Text-to-SQL Agent              │
  │     → 查 DuckDB                      │
  │     → 返回统计结果                     │
  │                                     │
  │  ② 语义查询                          │
  │     "博主们对冷暴力怎么看？"           │
  │     → RAG 检索（可按博主/分类过滤）    │
  │     → LLM 生成回答                    │
  │                                     │
  │  ③ 混合查询                          │
  │     "桃姐最近聊了什么？她关于吵架      │
  │      的建议是什么？"                   │
  │     → SQL 先查出桃姐最近视频列表       │
  │     → RAG 在这些视频中检索相关内容     │
  │     → LLM 融合生成回答                │
  └─────────────────────────────────────┘
```

## 四、统一数据层设计

### 4.1 DuckDB（结构化数据 · Text-to-SQL 查询）

```
video_meta 表
├── bvid          TEXT     主键
├── up_name       TEXT     UP主名称
├── up_uid        TEXT     UP主UID
├── title         TEXT     视频标题
├── publish_date  DATE     发布日期
├── category      TEXT     内容分类
├── duration      INT      时长（秒）
├── summary       TEXT     LLM生成摘要
├── tags          TEXT     标签（逗号分隔）
└── created_at    DATETIME 入库时间

up_info 表
├── uid           TEXT     主键
├── name          TEXT     UP主名称
├── total_videos  INT      视频总数
├── last_update   DATE     最近更新日期
└── config_file   TEXT     配置文件路径

query_log 表
├── id            INT      主键
├── question      TEXT     用户问题
├── route_type    TEXT     路由类型（structured/semantic/hybrid）
├── response_time INT      响应时间（ms）
└── created_at    DATETIME 查询时间
```

### 4.2 ChromaDB（向量数据 · RAG 检索）

```
collection: "video_knowledge"
├── document:     转写全文 / 摘要文本
├── embedding:    SiliconFlow Embedding 向量
└── metadata:
    ├── source:         "bilibili"
    ├── bvid:           视频BV号
    ├── up_name:        UP主名称
    ├── title:          视频标题
    ├── category:       内容分类
    ├── publish_date:   发布日期
    ├── chunk_index:    分块序号
    └── content_type:   "full" / "summary"
```

## 五、路由 Agent 设计

### 5.1 意图分类

```
路由Agent
├── 输入：用户自然语言问题
├── LLM 意图分析：
│   ├── structured   → 涉及数量统计、时间排序、比较等
│   ├── semantic     → 涉及观点、建议、内容理解等
│   └── hybrid       → 同时包含结构化和语义需求
├── 提取过滤条件：
│   ├── up_name      → 指定博主
│   ├── category     → 指定分类
│   ├── date_range   → 时间范围
│   └── keywords     → 关键词
└── 输出：路由决策 + 过滤条件
```

### 5.2 查询编排流程

```
用户问题 → 路由Agent
    ↓
  structured → Text-to-SQL Agent → DuckDB → 结果
  semantic   → RAG Pipeline → ChromaDB → LLM → 回答
  hybrid     → 并行：
               ├── Text-to-SQL → 结构化结果 + 过滤条件
               └── RAG（带metadata过滤）→ 语义结果
               ↓
             LLM融合 → 最终回答
```

## 六、Agent 协作架构

### 6.1 Text-to-SQL Agent（已有，不变）

```
意图理解Agent → Schema检索Agent → SQL生成Agent → SQL审查Agent
    ↓                ↓                 ↓               ↓
  解析问题       匹配表结构        生成SQL          校验+执行
```

### 6.2 RAG Agent（已有，增强 metadata 过滤）

```
用户问题 → Query改写 → 向量检索（+ metadata过滤）→ Reranking → LLM生成
                            ↑
                    ChromaDB collection
                    （带 up_name/category/date 过滤）
```

### 6.3 路由Agent（新增）

```
用户问题
    ↓
  Prompt：分析用户意图
    ↓
  判断类型：structured / semantic / hybrid
    ↓
  提取过滤条件（up_name, category, date_range）
    ↓
  调用对应子系统
    ↓
  收集结果 → LLM融合回答
```

## 八、部署架构

### 8.1 双环境部署

```
本地电脑（开发环境）                      NAS（生产环境）
AMD Ryzen 7 8845H · 24G · RTX 4060     Intel N150 · DDR4 8G
Docker Desktop（WSL2 后端）              Docker Compose 统一编排

                    ↕ GitHub（中心仓库） ↕

代码完全一致，环境差异走 .env + Docker profiles
下班后本地开发 → push → 上班时 NAS pull 并部署
```

### 8.2 NAS 生产环境（Docker Compose）

```
Docker Compose（DDR4 8G · Intel N150）
├── api-gateway        FastAPI 统一入口（路由Agent + 查询编排）  mem_limit: 512m
├── text-to-sql        Text-to-SQL Agent 服务                   mem_limit: 512m
├── rag-service        RAG 检索 + 生成服务                       mem_limit: 512m
├── bilibili-monitor   B站监控 Skill（Cron定时 · 按需启动）       mem_limit: 3g
├── chromadb           ChromaDB 向量数据库                       mem_limit: 1g
├── duckdb             DuckDB 结构化数据（挂载volume）
├── frontend           React 前端                                mem_limit: 256m
└── nginx              反向代理 + 域名                           mem_limit: 256m

内存分配：
├── 常驻服务：~3-4.5G（chromadb + FastAPI服务 + nginx）
├── bilibili-monitor：仅 Cron 时 docker compose run --rm 启动，跑完退出释放 3G
├── 转写峰值：~5.5-7.5G
└── NAS 支持模块化内存扩展，未来可升级

本地开发差异：
├── Whisper 使用 GPU 版（RTX 4060），不需要 faster-whisper
├── Docker Compose profiles 切换：--profile dev / --profile nas
└── 24G 内存，不需要 mem_limit 限制
```

## 九、技术栈汇总

```
数据采集：Python + B站WBI API + yt-dlp + faster-whisper (CPU · CTranslate2加速)
语音转写：faster-whisper small（推荐）或 Whisper medium
LLM精炼：Minimax M2.7 API
结构化存储：DuckDB
向量存储：ChromaDB + SiliconFlow Embedding
Agent框架：CrewAI（Text-to-SQL）+ 路由Agent
RAG框架：LangChain
后端：FastAPI + Python 3.11
前端：React 18 + TypeScript + Tailwind CSS
部署：Docker Compose（DDR4 8G · mem_limit 精细管理）
定时调度：Cron（每6小时 · bilibili-monitor 按需启动）
Agent平台：Hermes（B站监控Skill）
```

## 十、可查询示例

### 结构化查询（Text-to-SQL）
- "桃姐最近发了几个视频？"
- "哪个博主更新最频繁？"
- "最近一周有哪些视频？"
- "平均每个视频多长？"

### 语义查询（RAG）
- "博主们对冷暴力怎么看？"
- "吵架后不应该做什么？"
- "如何判断对方还在乎你？"
- "所有博主关于分手的建议"

### 混合查询（SQL + RAG）
- "桃姐最近聊了什么话题？她关于吵架的建议是什么？"
- "上周更新的视频中有没有提到复合的内容？"
- "更新最多的博主主要聊什么话题？"
