## 一、硬件环境

| 配置 | 本地电脑（开发） | NAS（生产） |
|------|-----------------|-------------|
| CPU | AMD Ryzen 7 8845H（8核16线程） | Intel N150（4核低功耗） |
| 内存 | 24GB | DDR4 8G（模块化，可扩展） |

```
开发策略：本地开发 + NAS 生产
├── GitHub 做中心仓库
├── 下班后在本地开发（RTX 4060 GPU 转写，24G 内存，调试快）

```

## 六、Docker Compose profiles

```yaml
# 开发环境（本地调试，含 GPU 转录服务）
docker compose --profile dev up -d

# 生产环境（NAS 部署，含定时调度）
docker compose --profile nas up -d

# 按需运行 bilibili-monitor
docker compose run --rm bilibili-monitor

# 手动触发（等效于 cron 调度）
docker compose run --rm bilibili-monitor python src/monitor_all.py

# 强制启用 ASR 转写
docker compose run --rm bilibili-monitor python src/monitor_all.py --asr
```

**profile 差异：**
- `dev`：所有服务 + gpu-service（GPU 转录），不含 bilibili-cron
- `nas`：所有服务 + bilibili-cron（每 6 小时自动调度），不含 gpu-service

## 八、ASR 设置与用量存储

```

{
  "enabled": false,                    ← 是否启用云 ASR
  "monthly_budget_minutes": 60,        ← 月度预算上限（分钟）

{
  "month": "2026-05",                  ← 当前统计月份
  "total_minutes": 23.5,               ← 本月累计转写时长
  "records": [                         ← 转写记录
    {
      "date": "2026-05-27",
      "up_name": "桃姐",
      "title": "视频标题",
      "duration_minutes": 8.2,
      "bvid": "BV1xxxxx",
      "cost": 0                        ← SenseVoiceSmall 免费
    }
  ]
}

读写权限：
├── router-agent：读写 settings.json + usage.json（API 端点）
├── bilibili-monitor：读取 settings.json + 写入 usage.json（转写时记录用量）
└── 共享卷：bilibili-data（Docker named volume）
```

## 九、性能预估（N150 CPU）

```
单条视频转写（8分钟音频）：
├── faster-whisper small（CPU · N150）：约 3-5 分钟
├── faster-whisper base（CPU · N150）：约 1-2 分钟
├── faster-whisper small（GPU · RTX 4060）：秒级
└── SiliconFlow ASR（SenseVoiceSmall · 云）：约 10-30 秒（网络上传+处理）

每次 Cron 触发（假设新增 3 个视频）：
├── 下载：约 2 分钟
├── 转写（本地 CPU）：约 10-15 分钟
├── 转写（云 ASR）：约 1-2 分钟（3 个视频 × 30秒）
├── 转写（GPU）：约 30 秒（3 个视频 × 10秒）
├── LLM 精炼：约 1 分钟（API 调用）
├── 入库：约 30 秒
└── 总计：约 5-20 分钟（取决于转写方式）

查询响应时间（实测）：
├── Text-to-SQL：3-30 秒（含 LLM 生成 SQL）
├── RAG 检索：3-16 秒（首次需构建 BM25 索引）
├── 混合查询：5-30 秒
└── 系统指标采集：~2.2 秒（Docker stats 并行采集优化后）
```

## 十、注意事项

```
⚠️ N150 转写时 CPU 占用较高
   → 转写期间暂停其他 CPU 密集型任务
   → 串行处理，不要并发（MAX_CONCURRENT_TRANSCRIBE=2）
   → 建议 NAS 侧使用云 ASR 替代本地 Whisper（减少 CPU 压力）

⚠️ B站 Cookie 会过期
   → 设置过期监控，失败时 QQ 通知提醒
   → 前端管理面板支持 Cookie 保存/删除/测试（B站 nav API 验证）
   → 采集触发前自动预检 Cookie 有效性

⚠️ 音频文件占空间
   → 转写完成后自动删除 m4a 文件
   → 只保留 txt 转写文本

⚠️ 云 ASR 成本控制
   → SenseVoiceSmall 模型免费，但仍需预算上限防止意外调用
   → 文件 > 50MB 自动跳过
   → 月度预算耗尽时自动停止转写
   → 前端可实时监控用量进度

⚠️ Docker 内存限制（DDR4 8G）
   → 常驻服务约 6.4G，在 8G 内可接受
   → bilibili-monitor 仅按需启动，跑完自动退出释放 4G
   → 各服务设置 mem_limit 防止单个服务占用过多
   → NAS 支持模块化内存扩展，未来可升级至 16G+

⚠️ Docker 镜像构建
   → 所有 Dockerfile 使用国内镜像加速（apt: 阿里云, pip: 清华 TUNA, npm: 淘宝）
   → 重建后清缓存已自动化：deploy.sh 收尾跑 `docker image prune -f`（清悬空旧版，禁用 `-a`）+ `docker builder prune -a -f`（构建缓存可全清）；手敲 `up -d --build` 由计划任务 `DockerAutoCleanup` 每周日 03:00 + 工作目录根 `docker-cleanup.bat` 兜底
   → bilibili-monitor 镜像最大（~2.12GB），其他较小

⚠️ 镜像迁移（避免重建）
   → 导出：docker save ai-rag ai-text-to-sql ai-router-agent ai-frontend ai-bilibili-monitor -o images.tar
   → 导入：docker load -i images.tar
   → 然后 docker compose --profile dev up -d（无需 --build）
```

## 十一、部署检查清单

```
新环境部署：
□ Docker + Docker Compose 已安装
□ git clone 项目代码
□ cp .env.example .env 并填入 API Key
□ docker compose --profile dev up -d --build
□ curl http://localhost:8090/api/stats 确认 RAG 正常
□ curl http://localhost:8000/health 确认 Router 正常
□ curl http://localhost:8000/api/up_info 确认 UP主管理正常
□ curl http://localhost:8000/api/asr/status 确认 ASR 管理正常
□ docker compose run --rm bilibili-monitor python src/migrate_refined.py
□ 前端访问 http://localhost 测试问答 + 管理面板

NAS 生产部署：
□ 上述全部步骤
□ docker compose --profile nas up -d（含 bilibili-cron）
□ 确认 bilibili-cron 容器运行：docker logs bilibili-cron
□ 管理面板 → 采集触发 → 配置 B站 Cookie
□ 管理面板 → 采集触发 → 测试采集
□ 管理面板 → GPU转录 → 配置 ASR 开关和预算（可选）

开发机 GPU 转录：
□ docker compose --profile dev up -d（含 gpu-service）
□ curl http://localhost:8011/api/gpu/status 确认 GPU 正常
□ 管理面板 → GPU转录 → 开始GPU转录
```
