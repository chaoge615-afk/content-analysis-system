# 智能内容分析系统 - 架构设计

> 最后更新：2026-05-27

---

## 一、系统定位

一套从**非结构化视频内容**到**智能问答**的端到端 AI 系统：
- 自动采集 B站博主视频内容
- 转写 → LLM 精炼 → 结构化入库 + 向量化入库
- 统一入口：路由 Agent 自动分发到 Text-to-SQL 或 RAG 知识库
- 支持结构化查询、语义检索、混合查询三种模式

## 三、数据流架构

### 3.1 数据采集链路（写入）

```
B站情感博主视频
    ↓ [Cron · 每6小时 · bilibili-cron 容器]
  WBI签名API → 拉取新视频列表
    ↓
  yt-dlp → 下载音频（m4a）
    ↓
  ┌─────────────────────────────────────────────────┐
  │  转写分支（按 ASR 设置自动选择）                   │
  │                                                   │
  │  ① 本地 Whisper（默认）                            │
  │     faster-whisper CPU/GPU → 语音转文字            │
  │                                                   │
  │  ② 云 ASR（硅基流动 SenseVoiceSmall · 免费）      │
  │     POST api.siliconflow.cn → 语音转文字           │
  │     预算控制：月度分钟数上限，超额自动停止          │
  └─────────────────────────────────────────────────┘
    ↓ [双层Checkpoint容错]
  DeepSeek API → 精炼生成视频摘要（三段式：核心观点+案例+建议）
    ↓
  ┌─────────────────────────────┐
  │        分流写入              │
  │                             │
  ├─────────────┬───────────────┤
  ↓             ↓               ↓
DuckDB        ChromaDB      文件系统
结构化元数据   向量存储        转写文本归档
  │             │               │
  │  video_meta │  embeddings   │  transcripts/
  │  up_info    │  + metadata   │  <分类>/xxx.txt
  └─────────────┴───────────────┘
```

### 3.2 数据查询链路（读取）

```
用户提问
    ↓
  路由Agent → 意图分析（MiniMax M2.7）
    ↓
  ┌─────────────────────────────────────┐
  │          三种路由策略                 │
  │                                     │
  │  ① 结构化查询                        │
  │     "桃姐最近发了几个视频？"           │
  │     → Text-to-SQL Pipeline           │
  │     → 查 DuckDB                      │
  │     → 返回统计结果                     │
  │                                     │
  │  ② 语义查询                          │
  │     "博主们对冷暴力怎么看？"           │
  │     → RAG BM25+向量混合检索           │
  │     → 可按 UP主/分类 metadata 过滤    │
  │     → DeepSeek/MiniMax LLM 生成回答   │
  │                                     │
  │  ③ 混合查询                          │
  │     "桃姐最近聊了什么？她关于吵架      │
  │      的建议是什么？"                   │
  │     → SQL 先查出桃姐最近视频列表       │
  │     → RAG 在这些视频中检索相关内容     │
  │     → LLM 融合生成回答                │
  └─────────────────────────────────────┘
```

## 四、统一数据层设计

### 4.1 DuckDB（结构化数据 · Text-to-SQL 查询）

```
video_meta 表
├── bvid          TEXT     主键
├── up_name       TEXT     UP主名称
├── up_uid        TEXT     UP主UID
├── title         TEXT     视频标题
├── publish_date  DATE     发布日期
├── category      TEXT     内容分类（31个情感分类）
├── duration      INT      时长（秒）
├── summary       TEXT     LLM生成摘要
├── tags          TEXT     标签（逗号分隔）
└── created_at    DATETIME 入库时间

up_info 表
├── uid           TEXT     主键
├── name          TEXT     UP主名称
├── total_videos  INT      视频总数
├── last_update   DATE     最近更新日期
└── config_file   TEXT     配置文件路径

query_log 表
├── id            INT      主键
├── question      TEXT     用户问题
├── route_type    TEXT     路由类型（structured/semantic/hybrid）
├── response_time FLOAT    响应时间（秒）
└── created_at    DATETIME 查询时间
```

### 4.2 ChromaDB（向量数据 · RAG 检索）

```
collection: "video_knowledge"
├── document:     转写全文 / 精炼摘要文本
├── embedding:    SiliconFlow Embedding 向量（BAAI/bge-large-zh-v1.5）
└── metadata:
    ├── source:         文件路径
    ├── bvid:           视频BV号
    ├── up_name:        UP主名称
    ├── category:       内容分类
    ├── chunk_index:    分块序号
    └── content_type:   "full" / "summary" / "chunk"
```

## 五、路由 Agent 设计

### 5.1 意图分类

```
路由Agent（router-agent :8000）
├── 输入：用户自然语言问题
├── LLM 意图分析（MiniMax M2.7 via OpenAI SDK）：
│   ├── structured   → 涉及数量统计、时间排序、比较等
│   ├── semantic     → 涉及观点、建议、内容理解等
│   └── hybrid       → 同时包含结构化和语义需求
├── 提取过滤条件：
│   ├── up_name      → 指定博主（仅明确提及时）
│   ├── category     → 指定分类（只能用 31 个有效分类名）
│   ├── date_range   → 时间范围
│   └── keywords     → 话题关键词（如"冷暴力""吵架"）
└── 输出：路由决策 + 过滤条件
```

**注意：** category 是目录分类名（如 `01_喜欢`），不是话题关键词。话题词放 keywords。

### 5.2 查询编排流程

```
用户问题 → 路由Agent
    ↓
  structured → Text-to-SQL Pipeline → DuckDB → 结果
  semantic   → RAG Pipeline → ChromaDB → LLM → 回答
  hybrid     → 并行：
               ├── Text-to-SQL → 结构化结果 + 过滤条件
               └── RAG（带metadata过滤）→ 语义结果
               ↓
             LLM融合 → 最终回答
```

### 5.3 强制路由

```python
# 跳过意图分类，直接指定路由
{"question": "...", "force_route": "rag"}   # 强制 RAG
{"question": "...", "force_route": "sql"}   # 强制 Text-to-SQL
```

## 六、各子系统架构

### 6.1 Text-to-SQL Pipeline

```
用户问题
    ↓
  Schema Agent → 匹配表结构（video_meta / up_info / query_log）
    ↓
  Intent Agent → 理解查询意图
    ↓
  SQL Gen Agent → 生成 DuckDB SQL
    ↓
  SQL Review Agent → 校验 + 执行
    ↓
  返回结果（JSON 表格）
```

**注意：** 声称用 CrewAI 但实际是纯 Python 手写 pipeline。

### 6.2 RAG Pipeline

```
用户问题
    ↓
  BM25 关键词检索（rank_cap=3）
    +
  向量相似度检索（top_k=5）
    ↓
  混合排序融合（HybridSearch）
    ↓
  metadata 过滤（up_name / category / bvid）
    ↓
  拼接上下文（附带来源信息）
    ↓
  LLM 生成回答
  ├── DeepSeek V4 Pro（默认，via OpenAI SDK）
  └── MiniMax M2.7（备选，via Anthropic SDK）
```

### 6.3 bilibili-monitor（数据采集）

```
monitor_all.py（入口）
    ↓
  扫描 config/*.yaml（UP主配置）
    ↓
  分批执行 monitor.py（MAX_CONCURRENT_TRANSCRIBE=2）
    ↓
  每个 UP 主：
  ├── get_up_videos.py → WBI API 拉取视频列表
  ├── download_videos.py → yt-dlp 下载 m4a
  ├── ┌────────────────────────────────────────┐
  │   │ 转写分支（按 ASR 设置选择）              │
  │   │  ① transcribe_local.py → faster-whisper │
  │   │  ② transcribe_asr.py → SiliconFlow ASR  │
  │   │     （预算检查 → multipart upload → 用量记录）
  │   └────────────────────────────────────────┘
  ├── refiner.py → DeepSeek API 精炼
  ├── db_writer.py → 写入 DuckDB
  ├── chroma_writer.py → 写入 ChromaDB
  └── qq_notify.py → QQ Bot 通知

ASR 设置存储（bilibili-data 卷）：
  .asr_config/settings.json → { enabled, monthly_budget_minutes, model }
  .asr_config/usage.json    → { month, total_minutes, records: [...] }
```

### 6.4 UP主管理（router-agent）

```
前端输入 B站链接
    ↓
  up_manager.parse_url() → 正则提取 UID 或 BVID
    ↓
  up_manager.fetch_profile() → B站 API 获取信息
  ├── BVID → GET /x/web-interface/view?bvid=xxx → owner.mid + owner.name
  └── UID  → GET /x/space/wbi/acc/info?mid=xxx
    ↓
  前端预览（名称、UID、头像）+ 选择 whisper_model
    ↓
  up_manager.add_up() → 生成 config/<uid>.yaml
    ↓
  下次采集自动包含新 UP主

API 端点：
  GET    /api/up_info/resolve?url=xxx  → 预览（不保存）
  POST   /api/up_info                  → 添加 { url, whisper_model? }
  GET    /api/up_info                  → 列表（含 DuckDB 视频数）
  DELETE /api/up_info/<uid>            → 删除
```

### 6.5 GPU 转录服务（开发机）

```
gpu-service 容器（Docker GPU 直通）
├── NVIDIA RTX 4060 · 8GB 显存
├── FastAPI :8011（仅开发机 localhost 可达）
├── 自动检测 CUDA GPU
└── 转写模式：
    ├── faster-whisper GPU（CUDA float16）→ 秒级转写
    └── CTranslate2 + cublas64_12.dll

前端 GpuTranscribe 组件：
├── GPU 状态卡片（名称、显存、PyTorch 版本）
├── 目录配置（downloads / transcripts）
├── 模型 + 设备选择（tiny/base/small/medium · CUDA/CPU）
├── 进度条 + 实时日志
└── 云 ASR 转写区域（NAS）
    ├── 开关（开启/关闭 ASR）
    ├── 月度预算（分钟数上限）
    ├── 用量进度条
    ├── 手动触发按钮
    └── 最近转写记录列表
```

## 十一、可查询示例

### 结构化查询（Text-to-SQL）
- "桃姐最近发了几个视频？"
- "哪个博主更新最频繁？"
- "最近一周有哪些视频？"
- "知识库有多少个视频？"

### 语义查询（RAG）
- "博主们对冷暴力怎么看？"
- "吵架后不应该做什么？"
- "如何判断对方还在乎你？"
- "如何追女生？"

### 混合查询（SQL + RAG）
- "桃姐最近聊了什么话题？她关于吵架的建议是什么？"
- "上周更新的视频中有没有提到复合的内容？"
- "更新最多的博主主要聊什么话题？"
