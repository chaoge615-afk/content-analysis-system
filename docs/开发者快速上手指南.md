## 1. 服务启动顺序与依赖

```
启动顺序（必须）：
  chromadb (8001)
    ↓
  text-to-sql (8010) + rag (8090)  [可并行，都依赖 chromadb]
    ↓
  router-agent (8000)              [依赖 text-to-sql + rag]
    ↓
  frontend (80)                    [依赖 router-agent]

按需运行：
  bilibili-monitor                 [依赖 chromadb]
  bilibili-cron                    [调度 bilibili-monitor，仅 nas profile]
```

**启动命令：**
```bash
# 开发环境
docker compose --profile dev up -d

# 生产环境（含定时调度）
docker compose --profile nas up -d

# 查看状态
docker compose --profile dev ps
```

## 2. 数据流与存储位置

### 写入链路
```
bilibili-monitor 写入：
  ├─ DuckDB: /app/data/content.db (volume: duckdb-data)
  │   ├─ video_meta 表 (视频元数据)
  │   ├─ up_info 表 (UP主信息)
  │   └─ query_log 表 (查询日志，由 router-agent 写入)
  └─ ChromaDB: video_knowledge collection
      ├─ content_type: "full" (转写全文)
      ├─ content_type: "summary" (精炼摘要)
      └─ content_type: "chunk" (分块内容)

text-to-sql 读取：DuckDB (只读)
rag 读取：ChromaDB (只读)
router-agent 读取：两者都读，写入 query_log
```

### 数据量级（当前状态）
- DuckDB: 1413 条视频元数据
- ChromaDB: 1760 个文档块（1413 文件分块后）
- 精炼文件: 1413 个 txt，31 个分类

## 4. 已知问题与解决方案

### 4.1 Windows 路径转换问题（MSYS/Git Bash）

**问题：** Git Bash 会将 `/app/source_data` 自动转换为 `D:/Git/app/source_data`

**解决方案：**
```bash
# 方案 1：禁用路径转换
MSYS_NO_PATHCONV=1 docker compose run --rm bilibili-monitor ...

# 方案 2：使用绝对 Windows 路径
docker compose run --rm bilibili-monitor python src/migrate_refined.py \
  --source "C:/Users/25022/Desktop/ai项目/relationship-analysis/references/情感素材库"
```

### 4.2 .env 文件 CRLF 编码问题

**问题：** Windows 下 `.env` 使用 CRLF 换行，`grep` + `cut` 提取的值带 `\r`，导致模型名变成 `deepseek-v4-pro\r`

**解决方案：**
```bash
# 转换换行符
dos2unix .env

# 或在读取时去除 \r
cat .env | grep "RAG_LLM_MODEL" | cut -d'=' -f2 | tr -d '\r'
```

**预防：** 使用 VS Code 时，右下角切换为 LF（不是 CRLF）

### 4.3 ChromaDB 向量检索返回空结果

**常见原因：**
1. **Embedding 模型语言不匹配**：使用中文模型 `BAAI/bge-large-zh-v1.5`，查询也必须是中文
   ```bash
   # ❌ 错误：英文查询
   "How to chase girls"
   
   # ✅ 正确：中文查询
   "如何追女生"
   ```

2. **分类器传了错误的 category filter**：
   - 检查 router-agent 日志，看 `filters` 字段
   - 有效分类只有 31 个（见 `intent_classifier.py`）
   - 话题词（如"冷暴力"）应该放 `keywords`，不是 `category`

3. **BM25 索引未构建**：首次查询会慢（构建索引）
   ```
   [混合检索] 构建 BM25 索引...
   [混合检索] BM25 索引构建完成，1760 个文档
   ```

### 4.4 MiniMax API 返回 ThinkingBlock

**问题：** MiniMax M2.7 返回的 `response.content` 包含 `ThinkingBlock` 对象，没有 `.text` 属性，直接访问会报错

**解决方案：**
```python
# ❌ 错误
content = response.content[0].text

# ✅ 正确：过滤只取 text 块
text_parts = [block.text for block in response.content if hasattr(block, 'text')]
content = "\n".join(text_parts) if text_parts else ""
```

### 4.5 Docker 容器内路径解析错误

**问题：** `Path(__file__).parent.parent.parent` 在容器内解析为 `/`（因为 WORKDIR 是 `/app`，脚本在 `/app/src/`）

**解决方案：**
```python
# 对于 /app/src/xxx.py，要获取 /app
ROOT_DIR = Path(__file__).parent.parent  # ✅ 正确
ROOT_DIR = Path(__file__).parent.parent.parent  # ❌ 错误，会到 /
```

### 4.6 ChromaDB Rust 后端 Windows 路径问题

**问题：** ChromaDB Rust 后端不支持路径含非 ASCII 字符（如中文）

**解决方案（已实现在 `chroma_writer.py`）：**
```python
# 临时切换到英文目录，使用相对路径
_original_cwd = os.getcwd()
_project_dir = str(Path(__file__).parent.parent)
try:
    os.chdir(_project_dir)
    _rel_path = os.path.relpath(persist_directory, _project_dir)
    self.client = chromadb.PersistentClient(path=_rel_path, ...)
finally:
    os.chdir(_original_cwd)
```

### 4.7 UP主名称不匹配导致查询空结果

**问题：** 用户使用简称/昵称（如"桃姐"、"安佳佳"），但数据库存储的是完整名称（"恋爱教头桃姐"、"是你的安佳佳呀"），导致 SQL 精确匹配失败。

**解决方案（已实现，三层层级）：**
1. **LLM 层**：`intent_classifier.py` 的 system prompt 末尾注入已知 UP主 名称列表，要求 LLM 将简称标准化为完整名称
2. **代码兜底层**：`_normalize_up_name()` 在 `_parse_response()` 中对 LLM 输出的 up_name 进行确定性匹配（精确匹配→包含匹配→子序列相似度匹配），无法匹配的名称转为 keywords
3. **查询注入层**：`query_dispatcher.py` 将标准化后的完整名称作为提示注入问题文本：`"问题（UP主完整名称是：恋爱教头桃姐）"`

**验证方法：**
```bash
# 测试名称标准化
curl -X POST http://localhost:8000/api/chat \
  -H "Content-Type: application/json" \
  -d '{"question": "桃姐关于冷战有什么建议？"}'
# 预期：route_type=hybrid，sql_result 非空，up_name 为 "恋爱教头桃姐"
```

**注意：**
- UP主 名称列表从 DuckDB `SELECT DISTINCT up_name FROM video_meta` 加载，缓存 5 分钟
- `SequenceMatcher` 阈值 0.6，低于此值的名称不会被标准化，转为 keywords
- 新增 UP主 后需等待缓存过期（5分钟），或重启 router-agent 容器

### 4.8 Cookie 注入失败：volume 名称不匹配

**问题：** 通过前端保存的 Cookie 文件在 router-agent 容器内可访问（`/app/data/bilibili_cookie.txt`），但由 Docker SDK 动态启动的 bilibili-monitor 容器找不到该文件，导致 `Cookie文件缺少必需字段` 错误。

**根因：**
- compose 实际创建的 volume 名带项目前缀（如 `ai_duckdb-data`）
- Docker SDK `client.containers.run(volumes={"duckdb-data": ...})` 创建的是**新的空 volume**（名 `duckdb-data`），与 compose 管理的 `ai_duckdb-data` 是**两个不同的 volume**
- 容器内 `/app/data/bilibili_cookie.txt` 指向空的 volume，文件不存在

**解决方案（已实现）：**
- `monitor_trigger._build_env()` 改为直接读取 Cookie 文件**内容**，作为 `BILIBILI_COOKIE` 环境变量传入容器
- `bilibili-monitor/src/cookie_utils.py` 检测到 `BILIBILI_COOKIE` 环境变量时，将内容写入容器内文件再解析
- 不再依赖 volume 路径一致性

**教训：**
- Docker SDK 动态启动容器时，**不要依赖 named volumes**（名称可能带前缀）
- 传递配置数据优先用**环境变量内容**而非文件路径

## 6. 性能优化要点

### 6.1 RAG 检索性能

- **首次查询慢**：BM25 索引构建（1760 文档约 2-3 秒），后续查询快
- **chunk_size=500**：适合短文本（精炼摘要），长文本建议 1000
- **top_k=5**：默认返回 5 个最相关文档，可在 `.env` 调整

### 6.2 bilibili-monitor 性能

- **Whisper 转写**：CPU 模式 `small` 模型约 1-2 分钟/视频，GPU 快 10 倍
- **并发限制**：`monitor_all.py` 默认 `MAX_CONCURRENT_TRANSCRIBE=2`，避免 OOM
- **批量写入**：ChromaDB 每 20 个文档一批，DuckDB 每条单独写入

### 6.3 Docker 资源限制

```yaml
# docker-compose.yml 中各服务 mem_limit
chromadb: 1g
text-to-sql: 2g
rag: 2g
router-agent: 1g
frontend: 256m
bilibili-monitor: 4g
bilibili-cron: 128m
```

NAS 总内存 8GB，当前配置约 10.4GB（有重叠，实际运行约 6-7GB）

### 6.4 API 接口性能

**原则：简单数据查询不要走 LLM 全链路。**

| 接口 | 优化前 | 优化后 | 方法 |
|------|--------|--------|------|
| `/api/system_metrics` | 23s | 2.2s | SQL统计改 DuckDB 直查 + Docker stats 并行采集 |
| `/api/up_list` | 25ms → 2ms | DuckDB 连接复用（全局 `_get_db()`） |
| `/api/recent` | 30-90s → 10ms | 从 LLM text-to-sql 改为 DuckDB 直查 |
| `/api/categories` | 30-90s → 10ms | 同上 |

**Docker stats 并行化模式：**
```python
# 每个容器 stats(stream=False) 阻塞 ~1.5s（采集两个 CPU 样本）
# 5 个容器串行 = 8s，并行 = 1.5s
with concurrent.futures.ThreadPoolExecutor(max_workers=len(targets)) as ex:
    futures = {ex.submit(_collect_one, c): c for c in targets}
    for fut in concurrent.futures.as_completed(futures):
        name, info = fut.result()
        metrics[name] = info
```

**DuckDB 连接复用：**
- router-agent 全局维护一个只读连接 `_db_conn`（惰性初始化）
- 避免每次请求 connect/close 的 ~20ms 开销
- 写入连接由各服务独立管理（text-to-sql、bilibili-monitor）

## 8. 故障排查清单

### 服务启动失败

- [ ] 检查端口占用：`netstat -ano | findstr :8000`
- [ ] 检查环境变量：`docker compose config` 看是否有未设置的变量
- [ ] 检查依赖服务：`docker compose --profile dev ps` 看 chromadb 是否 healthy
- [ ] 查看启动日志：`docker logs <service>`

### 问答返回空结果

- [ ] **直调 RAG**：`curl -X POST http://localhost:8090/api/ask_video -d '{"question": "..."}'`
  - 有结果 → 问题在 router-agent
  - 无结果 → 问题在 RAG 或 ChromaDB
- [ ] **检查 ChromaDB 数据**：`curl http://localhost:8090/api/stats` 看 `video_chunks` 是否为 0
- [ ] **检查查询语言**：Embedding 是中文模型，查询必须是中文
- [ ] **检查 filters**：router-agent 日志看是否传了不存在的 category

### bilibili-monitor 运行失败

- [ ] **Cookie 过期**：登录 B站，F12 → Network → 找 Cookie，更新 `.env`
- [ ] **API 限流**：B站 API 有频率限制，加 `--dry-run` 测试
- [ ] **磁盘空间**：视频下载需要空间，检查 `docker system df`
- [ ] **ChromaDB 连接**：确认 `CHROMA_HOST=chromadb` 且 chromadb 容器 healthy

### 构建镜像失败

- [ ] **网络问题**：国内镜像配置在 Dockerfile 中，检查是否被注释
- [ ] **依赖冲突**：查看 `pip install` 错误信息，可能是版本不兼容
- [ ] **磁盘空间**：`docker builder prune -a -f` 清理缓存
- [ ] **Docker 版本**：需要 Docker 20.10+（支持 BuildKit）
